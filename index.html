⚡ QPS (Queries Per Second) — 每秒查询数

衡量系统每秒能处理多少"读"请求。比如用户打开首页，浏览器发起 3 个 GET 请求（HTML + CSS + JS），这就是 3 次查询。QPS 反映的是查询吞吐。

💼 TPS (Transactions Per Second) — 每秒事务数

衡量每秒完成的"业务操作"数。一个事务是一个完整的业务闭环：比如"下单"= 验证用户 → 查库存 → 锁库存 → 创建订单 → 写流水 → 发通知，这一串操作才算 1 个 TPS。

👥 并发数 — 同时在线的用户数

注意区分"并发连接数"和"并发请求数"。1000 个用户在线不代表 1000 个同时在请求。通常 QPS ≈ 并发数 / 平均响应时间。

场景QPSTPS典型系统
搜索10万+—Elasticsearch
商品详情页5万—Nginx + Redis
下单5千1千MySQL + MQ
支付2千500银行接口 + 分布式事务
💡 面试金句：「TPS 衡量的是业务吞吐，一个 TPS 背后可能是 5-10 次数据库操作。所以 QPS 通常比 TPS 高一个数量级。做性能评估时，先定 TPS 目标，再反推 QPS。」

为什么不用平均值？假设 100 个请求，99 个在 50ms 完成，1 个因为 Full GC 卡了 5 秒。平均值 = (99×50 + 5000)/100 ≈ 100ms，看起来"还行"。但实际有 1% 的用户等了 5 秒——体验极差！

⚠️ 面试陷阱：面试官说"我们系统平均响应时间 100ms，性能没问题"→ 你要立刻追问：「P99 是多少？长尾延迟怎么处理？」
指标含义通俗理解典型 SLA
P5050% 请求的响应时间"一半用户"的体验<50ms
P9595% 请求的响应时间"绝大多数"用户体验上限<200ms
P9999% 请求的响应时间SLA 承诺底线，超出=故障<500ms
P99999.9% 长尾延迟极端抖动，排查 GC/锁竞争<2s
💡 面试金句：「我们 SLA 定在 P99 < 500ms。P999 如果突然飙升到 3s+，第一反应是查 Full GC 日志和 数据库锁等待，这两个是长尾延迟最常见的元凶。」

📋 标准压测流程（6步法）
1

定目标

明确 SLA：响应时间 P99<500ms、TPS≥1000、错误率<0.1%。没有目标=没有标准。

2

设计场景

模拟真实流量：70% 浏览 + 20% 搜索 + 10% 下单。覆盖峰值 ×2 的极端场景。编写压测脚本。

3

准备环境

环境需与生产 1:1 配置。隔离压测集群，做数据脱敏。预热完成后开始。

4

阶梯加压

10%→50%→100%→130%→150%，每阶段 15-30 分钟。记录各阶段的 QPS、延迟、错误率。

5

监控分析

CPU/内存/磁盘 IO/网络/DB连接池/JVM GC。用 Grafana+Prometheus 实时监控。

6

瓶颈修复

定位 Top 5 慢请求 → DB 索引 → 连接池 → 缓存 → 代码优化。修复后 回归压测验证。

分布式服务器

CAP定理 · 一致性协议 · 服务发现 · 负载均衡 · 分布式事务 · 一致性哈希

CAPRaft一致性哈希ZooKeeper2PC/TCCNginxRedis Cluster幂等性

CAP：Consistency（一致性）· Availability（可用性）· Partition Tolerance（分区容错）。

在分布式系统中，网络分区 (P) 无法避免——网线可能断、交换机可能坏、机房可能停电。所以 P 是必须接受的现实，你只能在 C 和 A 之间做权衡。

💡 通俗理解：就好比异地恋——要么每天打电话确认对方在家（CP，强一致但可能打不通），要么发微信不管对方看没看到（AP，随时可用但可能消息丢了），你没法同时保证"随时联系"和"信息完全同步"。
选择典型系统行为适用场景
CPZooKeeper, etcd, Redis Cluster分区时拒绝写入，保一致性配置中心、分布式锁、选主
APEureka, Cassandra, DynamoDB分区时允许写入，后续修复不一致注册中心、日志、社交Feed
💡 面试金句：「CAP 不是非黑即白的选择题，而是权衡程度。关键是根据业务场景选择：钱的事用 CP（支付、配置中心），体验的事用 AP（注册中心、推荐系统）。」
❌ 取模的问题

假设 3 台缓存服务器，用 hash(key) % 3 分配。增加第 4 台 → 75% 的 key 会映射到不同服务器，大量缓存失效（缓存雪崩）。

✅ 一致性哈希的解法
  1. 把服务器 IP 哈希后映射到 0~2³² 的哈希环上
  2. key 也哈希到环上，顺时针找到第一个服务器节点
  3. 加机器：只影响新机器到下一台之间的 key（约 1/N 的数据迁移）
  4. 减机器：只影响被删节点覆盖的 key（约 1/N 的数据迁移）
💡 面试金句：「一致性哈希解决的是动态扩缩容时缓存大量失效的问题。哈希环 + 虚拟节点（每台物理机器映射 100-200 个虚拟节点到环上）还能避免数据倾斜。」
场景：下单扣库存

用户下单时，需要同时做两件事：① 订单服务创建订单 ② 库存服务扣减库存。这两个操作在不同的数据库上——没法用 MySQL 事务。

方案原理一致性性能复杂度场景
2PC协调者 → 所有参与者 Prepare → 全 OK 则 Commit强一致差（同步阻塞）高银行转账
3PC2PC + 预提交 + 超时机制强一致稍好更高几乎不用
TCCTry(预留) → Confirm(确认) → Cancel(取消)强一致好高（业务侵入）订单/支付
Saga长事务拆成多个本地事务+补偿最终一致很好中长流程业务
MQ消息本地事务 + 消息表，异步消费最终一致最好低大多数场景
💡 面试金句：「大多数场景用 本地消息表 + MQ 就够了（最终一致、高吞吐）。强一致场景（金额、库存扣减）才上 TCC。2PC 理论知道但实践中很少用，性能太差。」
方案实现可靠性性能死锁防护
RedisSETNX key value EX 30 + Lua 解锁中（主从切换可能丢锁）极高过期时间 + RedLock
ZooKeeper临时顺序节点 + Watch高（CP 系统）中会话断开自动删节点
数据库乐观锁UPDATE ... WHERE version=N高低超时清理任务
python
import redis, uuid, time
r = redis.Redis(host='localhost', port=6379)

class RedisLock:
    def __init__(self, key, expire=30):
        self.key = key; self.val = uuid.uuid4().hex; self.expire = expire

    def acquire(self):
        # SETNX + EXPIRE 原子化
        return r.set(self.key, self.val, nx=True, ex=self.expire)

    def release(self):
        # Lua 脚本保证原子性：只删自己持有的锁
        script = """
        if redis.call('get', KEYS[1]) == ARGV[1] then
            return redis.call('del', KEYS[1])
        else return 0 end"""
        r.eval(script, 1, self.key, self.val)

服务器容灾恢复

RTO/RPO · 多活架构 · 故障转移 · 混沌工程 · 熔断降级限流 · 备份策略

RTO/RPO主备切换双活多活Chaos MeshSentinel备份策略

RTO (Recovery Time Objective)：恢复时间目标——故障发生后，多久能恢复服务？

RPO (Recovery Point Objective)：恢复点目标——恢复后，最多丢失多少数据？（时间维度的数据量）

💡 通俗理解：RTO 是"你能忍多久不能用"，RPO 是"你能忍丢多少数据"。核心支付RTO=0（秒级切换），RPO=0（零丢失），代价是每年几百万的异地多活架构。
业务RTORPO典型方案成本
核心支付<30s0同城双活+异地灾备+实时同步极高
电商下单<5min<30s主备切换+MQ补偿高
游戏战斗<1min<1min主备+Redis持久化中
数据分析<4h<1h每日备份+冷备恢复低
L1

冷备 (Cold Standby)

定期备份到异地。故障时人工介入恢复。RTO=小时级，成本最低。

L2

温备 (Warm Standby)

备机已配置、数据异步同步。故障时半自动切换。RTO=分钟级。

L3

主备 (Active-Passive)

主备实时同步，通过 Keepalived/VRRP 做 VIP 漂移。故障时自动切换 <30s。

L4

双活 (Active-Active)

两机房同时服务，流量按比例分配（如 6:4），任一机房故障自动切流。

L5

异地多活

多 Region 同时服务，数据异步最终一致。需要解决冲突合并、时钟同步等难题。

机制通俗理解触发条件动作工具
熔断断路器跳闸错误率 > 50%直接拒绝请求，不再调下游Sentinel, Resilience4j
降级保命模式系统负载高 / 依赖不可用返回兜底数据 / 关非核心功能业务逻辑实现
限流守门员QPS 超过阈值拒绝 / 排队 / 降级处理Sentinel, RateLimiter
💡 面试金句：「限流是预防（控制入口流量），熔断是止损（下游挂了不拖累上游），降级是保命（系统快崩了先保核心链路）。三者配合：先限流→扛不住就降级→依赖出问题就熔断。」

核心思想：与其等生产环境出问题，不如主动注入故障，在生产中验证系统的容错能力。

  • 故障注入类型：Pod Kill | 网络延迟 | DNS 故障 | 磁盘满 | CPU 打满
  • 工具：Chaos Mesh, LitmusChaos, Gremlin
  • 原则：先小后大（控制爆炸半径）、可观测、可回滚

本质：装饰器是一个接受函数作为参数、返回新函数的高阶函数。Python 的 @ 语法糖让它可以优雅地"包装"函数。

python
# 1. 最简单的装饰器：计时
import time
def timer(func):
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"{func.__name__} 耗时: {time.time()-start:.3f}s")
        return result
    return wrapper

@timer
def slow_api(): time.sleep(1)

# 2. 带参数的装饰器：重试
def retry(times=3, delay=1):
    def decorator(func):
        def wrapper(*args, **kwargs):
            for i in range(times):
                try: return func(*args, **kwargs)
                except Exception as e:
                    if i == times - 1: raise
                    time.sleep(delay)
        return wrapper
    return decorator

@retry(times=5, delay=0.5)  # 失败重试5次，间隔0.5s
def call_external_api(url): return requests.get(url)

# 3. 面试常问：保留原函数元信息
from functools import wraps
def my_decorator(func):
    @wraps(func)  # 保留 func.__name__, func.__doc__
    def wrapper(*args, **kwargs): return func(*args, **kwargs)
    return wrapper
💡 面试金句：「装饰器的核心是闭包。@wraps 很重要，否则装饰后的函数会丢失原名和文档。用装饰器可以无侵入地给函数添加日志、计时、权限校验、重试等横切关注点。」

生成器是用 yield 关键字的函数，调用时不立即执行，而是返回一个迭代器对象。每次 next() 执行到下一个 yield，状态暂停在当前位置。

python
# 普通方式：一次加载100万条 → OOM
def read_all(filename):
    return open(filename).readlines()

# 生成器方式：每次只读一行 → 内存占用极低
def read_large_file(filename):
    with open(filename) as f:
        for line in f:
            yield line.strip()

# 列表推导式 → 生成器表达式（只改一个符号）
nums = [x*x for x in range(10000000)]  # 列表：占 ~80MB 内存
nums = (x*x for x in range(10000000))  # 生成器：几乎不占内存

# yield from: 委托子生成器
def chain(*iterables):
    for it in iterables:
        yield from it  # 等价于 for x in it: yield x
💡 面试金句：「处理大数据时能生成器就不用列表。生成器的优势是惰性计算：不一次性加载全部数据，而是按需生成。配合 itertools 可以实现高效的数据流水线。」

GIL (Global Interpreter Lock)：CPython 解释器中的全局锁，保证同一时刻只有一个线程执行 Python 字节码。这意味着多线程在 CPU 密集型任务上无法利用多核。

场景用什么原因示例
CPU 密集型多进程 (multiprocessing)每个进程有独立 GIL图像处理、科学计算
IO 密集型多线程 / asyncioIO 等待时释放 GILWeb 爬虫、API 调用
高并发 Webasyncio (协程)单线程事件循环，无 GIL 竞争FastAPI, aiohttp
💡 面试金句：「GIL 是 CPython 的设计权衡——简化了内存管理，代价是多核 CPU 密集型计算受限。IO 密集型用多线程（GIL 在 IO 等待时释放），CPU 密集型用多进程或 C 扩展。Python 3.13 开始有可选的无 GIL 模式。」

浅拷贝（copy.copy()）：只复制对象本身，内部嵌套对象共享引用。

深拷贝（copy.deepcopy()）：递归复制所有嵌套对象，完全独立。

python
import copy
a = [[1, 2], [3, 4]]

b = copy.copy(a)     # 浅拷贝：b 是新列表，但内部子列表还是同一个
c = copy.deepcopy(a) # 深拷贝：c 及其所有子对象全是新的

a[0][0] = 99
print(b[0][0])  # 99 ← 受影响！(浅拷贝共享内部引用)
print(c[0][0])  # 1  ← 不受影响 (深拷贝完全独立)

# 常见浅拷贝方式（都是浅拷贝！）
x = lst[:]           # 切片
x = list(lst)        # 构造函数
x = lst.copy()       # copy 方法

Python 使用引用计数 + 标记清除 + 分代回收三重机制管理内存：

  • 引用计数：每个对象维护一个引用计数。计数=0 时立即回收。sys.getrefcount(obj) 查看。
  • 标记清除：处理循环引用（A 引用 B，B 引用 A，但外部都无法访问它们）。
  • 分代回收：对象分 0/1/2 三代。新生对象在 0 代，经历回收次数越多越"老"，回收频率越低。
⚠️ 常见内存泄漏：全局变量持有大对象、循环引用 + __del__、C 扩展未正确释放。

# api_client.py — 接口层：封装所有 HTTP 调用
import requests

class ApiClient:
    def __init__(self, base_url, token=None):
        self.base = base_url
        self.session = requests.Session()  # 复用连接，更快
        if token: self.session.headers.update({
            "Authorization": f"Bearer {token}"})

    def login(self, user, pwd):
        return self.session.post(f"{self.base}/login",
            json={"username": user, "password": pwd})

    def get_item(self, item_id):
        return self.session.get(f"{self.base}/items/{item_id}")

    def create_order(self, item_id, qty):
        return self.session.post(f"{self.base}/orders",
            json={"item_id": item_id, "qty": qty})

# test_e2e.py — 业务层：编排接口调用，模拟用户操作
def test_order_flow(api):
    # 1. 登录
    r = api.login("test_user", "test123")
    assert r.status_code == 200

    # 2. 查看商品
    r = api.get_item(42)
    assert r.json()["price"] > 0

    # 3. 下单
    r = api.create_order(42, 1)
    assert r.status_code == 201
    return r.json()["order_id"]

Page Object 模式：将每个页面封装成一个类，页面操作 → 类方法，页面元素 → 类属性。好处：UI 变了只改一处。

python
from selenium import webdriver
from selenium.webdriver.common.by import By
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

class LoginPage:
    def __init__(self, driver):
        self.driver = driver
        self.username = (By.ID, "username")
        self.password = (By.ID, "password")
        self.login_btn = (By.XPATH, "//button[@type='submit']")

    def login(self, user, pwd):
        self.driver.find_element(*self.username).send_keys(user)
        self.driver.find_element(*self.password).send_keys(pwd)
        self.driver.find_element(*self.login_btn).click()
        return HomePage(self.driver)

class HomePage:
    def __init__(self, driver):
        self.driver = driver
        self.welcome = (By.CSS_SELECTOR, ".welcome-msg")

    def get_welcome_text(self):
        el = WebDriverWait(self.driver, 10).until(
            EC.presence_of_element_located(self.welcome))
        return el.text

def test_login():
    driver = webdriver.Chrome()
    driver.get("https://example.com/login")
    home = LoginPage(driver).login("admin", "test123")
    assert "欢迎" in home.get_welcome_text()
    driver.quit()
